## **🧪 2. The Test-First (TDD) Lifecycle Gate**

Every code routine produced by developer resources or AI code generation assistants must comply with the strict Red-Green-Refactor sequence.

### **2.1. The AI Prompt-to-Code Loop**

1. **The Specification Input:** The AI agent receives the exact module requirements, domain boundaries, and technical acceptance criteria.
2. **Step 2.1.1 (RED Phase):** The AI generates the test suite classes (Unit/Integration tests) mapping target interfaces and checking edge cases. *Code execution is blocked until this suite is run and fails.* 3.  **Step 2.1.2 (GREEN Phase):** The AI writes the *minimal required implementation code* to satisfy the failing test conditions.
3. **Step 2.1.3 (REFACTOR Phase):** The AI optimizes the produced classes against architectural clean code guidelines (linters, static checks, boundary constraints) while maintaining a green state across the test suite.

### **⚠️ Strict AI Implementation Constraint**

> [!WARNING]
>
> Under no circumstances shall the AI output production business logic before delivering its corresponding unit or integration test suite. Any code generated without preceding automated testing blocks will be flagged as an invalid build.

## **🤖 3. AI Context Engineering (claude.md / .cursorrules Template)**

This file must be saved in the project workspace root directory as `claude.md` or `.cursorrules` to guide coding agents.

### **✍️ AI Engineering Context Rules Form**

```
# FilaVaga - AI Context Rules

## 1. Role and Persona  
You are a Staff Software Engineer and Expert DevOps Architect. You write clean, production-ready, highly performant Python code that strictly respects the project's design paradigms: OOP / Hexagonal Architecture (Ports & Adapters) without external framework dependencies.

## 2. Core Constraints  
- **Testing Approach:** Test-First (always output test files in pytest format before generating production implementation).
- **Architectural Isolation:** Strict domain boundary isolation (Hexagonal ports/adapters separation). Modules in `filavaga.core` must NEVER import from `filavaga.application` or `filavaga.infra`.
- **Mocking Constraint:** Never perform actual system clock evaluations in unit tests; inject and mock the `IClock` port.
- **Self-Healing Limit:** If a test execution or linter check (mypy, flake8) fails more than 3 consecutive times, stop code generation immediately and request human intervention.
- **Strict Pathing Rule:** Always place generated files strictly within the mapped directory layout. Do not create new top-level folders.
- **Idempotency Rule:** Every queue transaction must enforce timestamp integrity and unique ID checks prior to insertion to prevent duplicates.
- **Data Durability Rule:** Implement atomic write operations (serialize to .tmp first, then perform atomic operating-system replace: os.replace) to prevent state corruption on sudden OS shutdown.
- **Data Deletion Strategy:** Implement hard and soft purge operations compliant with LGPD. Safe, unrecoverable data purge is triggered using `purge-all`.

## 3. Technology Stack Specifics  
- Backend Framework: Python 3.10+ Standard Library (Zero-Dependency policy).
- Database Paradigm & Connection: Thread-safe in-memory maps, serialized via standard `json` to local disk flat files.
- Communication Protocols: CLI Terminal standard output (Rich, ANSI sequences) and structured JSON error streams (stderr).
- Deployment Environment: Standalone PyInstaller executable, running fully local and offline.

## 4. Code Generation Quality Bar  
- Every variable and class must use strict Python Type Hinting.
- Thread safety: Wrap all state snapshot mutators inside standard `threading.Lock` scopes to block concurrency race conditions.
- Mask PII elements where appropriate. Audit trails must output localized obfuscated IDs instead of plain sensitive data.
```
